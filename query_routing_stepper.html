1
Date pattern detection
Regex \d{1,2}/[A-Za-z]{3}/\d{4} extracts the date string (e.g. 15/Jan/2024) from the query.
2
Date index cache check
If _date_index_cache is populated (from a previous date query this session), it is reused instantly. If not, the full collection scan runs once and is cached.
3
Full collection scan
Every record's Created, Updated, and page content is checked for the target date string. Matching issue_key values are collected.
4
Return matched keys
Matched issue keys are returned directly as a sorted list — no LLM call needed for this path.
Result: Sorted list of issue keys that had activity on the given date. Fast after first call due to in-memory caching.
e.g. "List all In Progress tickets" or "Show High priority bugs" or "How many Fixed vs Duplicate tickets?"
1
Structured filter detection
detect_structured_filter() checks the query for known values across 4 dimensions — Status, Priority, Resolution, Severity — using word-boundary regex to avoid false matches (e.g. "High" inside "Highest").
2
Exact count injection
For each matched value, vectorstore.get(where=single_filter) fetches the real count from the full dataset. These counts are prepended to the LLM context so it sees actual numbers, not estimates.
3
Chroma $where filter search
vectorstore.similarity_search(query="", filter=filter_dict, k=200) — pulls up to 200 matching tickets directly via metadata, sorted by most recently updated.
4
Top 25 + LLM extraction prompt
The 25 most recently updated tickets are passed to Gemini with the extraction mode prompt. The exact counts prefix ensures the LLM answers count/comparison questions accurately.
Result: Accurate ticket lists and counts for status/priority/resolution/severity queries — no fuzzy matching, no hallucinated numbers.
e.g. "Show unresolved bugs with low priority" or "Find tickets about authentication failures"
1
Multi-query expansion
Gemini rewrites the original query into 3 reformulated search queries focused on unresolved tickets, backlog spillover, pending tasks — broadening recall.
2
Hybrid search per query
For each expanded query: Dense retrieval (ChromaDB top 20, weight 0.7) + Sparse retrieval (BM25 top 20, weight 0.3) are combined via reciprocal rank fusion. Top 40 docs returned per query.
3
Dynamic Top-K filtering
Docs below score threshold (0.2 for general, 0.05 for analytical) are dropped. Max 10 kept per query. All results are pooled across the 3 queries.
4
Deduplication
Duplicate docs (same page_content) are removed using a dict keyed on content. Only unique docs proceed.
5
Cross-encoder reranking
BAAI/bge-reranker-base scores every (query, doc) pair. Top 5 highest-scoring docs are selected for the final context.
6
LLM extraction prompt
Top 5 docs passed to Gemini with strict "answer only from context" extraction prompt.
Result: Semantically relevant tickets retrieved even when query words don't appear verbatim in issue descriptions.
e.g. "Why are so many tickets stuck in Needs Triage?" or "Give me an analysis of high severity bugs"
1
Analytical intent detection
Keywords why, reason, analysis in the query trigger analysis mode. Structured filter is skipped entirely — even if status/priority words are present.
2
Multi-query + hybrid search (lower threshold)
Same hybrid search path as semantic, but Dynamic Top-K threshold drops to 0.05 — casts a wider net to gather more contextual evidence for the analysis.
3
Cross-encoder reranking → top 5
Same reranking step as semantic path.
4
Anti-hallucination analysis prompt
Gemini is instructed to analyse using ONLY the 8 available fields (Issue Key, Type, Status, Priority, Resolution, Severity, Created, Updated, Description). If the question asks about sprints, velocity, or story points — not tracked fields — the model must say so explicitly instead of inferring.
Result: Grounded pattern analysis — root cause, trends, frequency — without fabricating data that isn't in the dataset.
